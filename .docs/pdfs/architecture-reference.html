System at a Glance

LlamaTrade is a multi-tenant SaaS for algorithmic trading. Users build (or pick) strategies in a visual builder, backtest them against historical data, and run them live or in paper mode against their own Alpaca brokerage account. We never custody funds — Alpaca is the broker of record; we are the automation layer.

The shape of the system

It is a set of focused microservices. The browser talks directly to each service over the Connect protocol (no API gateway); services talk to each other over gRPC; and asynchronous / real-time work flows through Redis. One service — Portfolio — owns the event-sourced ledger that is the book of record for money and positions.

  • 10 services (9 backend + web), ports 88xx
  • Connect (HTTP/1.1+JSON) browser→service; gRPC (HTTP/2) service→service
  • Redis for Celery jobs, live streams, and the ledger fill bus
  • Postgres 16 + TimescaleDB (bars); per-tenant Row-Level Security
  • Proto is the single source of truth for contracts & enums
  • BYO Alpaca keys, encrypted per tenant (AES-256-GCM)

Core design principles

The ledger is the book of record
Event-sourced, double-entry; per-sleeve positions, cash & P&L are pure folds. Conservation is a checksum, so drift is never silent.
Proto is the single source of truth
Contracts & enums defined once, generated for Python + TypeScript. Integers internally; strings only at external boundaries.
Durability layered by authority
The broker owns live positions, the ledger owns money & P&L, the executor owns order intent — no single session log to keep in sync.
Idempotent by construction
A deterministic client_order_id gives exactly-once submission and doubles as the ledger's attribution key on every fill.
Isolated by default
tenant_id on every row, Row-Level Security as a backstop, and X-Tenant-ID propagated across services.
Bring your own brokerage
Users connect their own Alpaca account; LlamaTrade automates it but never custodies funds — Alpaca is the broker of record.

Three transport lanes, each for a job. Contracts are shared via Protocol Buffers; make proto generates Python and TypeScript from libs/proto.

Proto is the source of truth

Enums are defined once in proto and generated for both languages — integers internally and for storage, strings only at external boundaries (Alpaca, display).

enums — integers internally, strings only at the edgefrom llamatrade_proto.generated import ORDER_SIDE_BUY, ORDER_STATUS_FILLED from src.models import order_side_to_str # int → "buy"/"sell" for Alpaca if order.side == ORDER_SIDE_BUY: # integers internally await alpaca.submit_order(side=order_side_to_str(order.side)) # strings only at the boundary

Why no API gateway decided

LlamaTrade uses direct service communication via Connect rather than a gateway. The rationale, for a frontend-only API with <15 services and strong shared-library discipline:

  • No SPOF — a gateway down means all APIs down; per-service auth failures stay isolated.
  • Lower latency — no proxy hop (+1–5ms), important for real-time data.
  • Native streaming — Connect streams without gRPC-Web transcoding.
  • Independent scaling — each service scales to its own load.
  • Simpler local dev — no gateway process to run.
  • Consistency via llamatrade_common shared middleware (auth, logging, errors).

Revisit if we expose a public third-party API, need per-key quotas, or grow past ~20 services — services already own their auth, so a gateway would layer on rather than replace.

4.2 — Live trading runner (two clocks, four loops)

Deploying a strategy starts a runner bound to the tenant's Alpaca credentials. It ticks on a 1-minute bar clock, but a date-based rebalance gate (the :rebalance frequency, ≤ once/day) decides when target weights recompute.

The runner loop — fills are the source of truth for positions
1
Data in 1-min bars
Runner consumes the live bar stream from Market Data and ticks every minute the market is open.
2
Strategy logic rebalance gate
On a qualifying bar (gate open ≤ once/day) the compiled strategy recomputes target weights for its sleeve; otherwise it holds.
3
Risk + circuit breaker
Sleeve-aware risk checks (buying power, position/order/daily-loss limits, rate); consecutive-loss / drawdown circuit breaker can halt the session.
4
Order out deterministic id
One rebalance can emit many orders (one per symbol whose holding changes). Executor submits to Alpaca immediately (market/limit/stop/stop-limit/bracket) with a deterministic client_order_id for exactly-once submission.
5
Fills in trade_updates
A trade_updates WebSocket delivers fills asynchronously — submission ≠ instant fill. Fills are the source of truth for positions, not optimistic local state.
6
Reconcile + book
Every ~5 min the runner reconciles local positions against the broker and corrects drift; terminal fills are published to the portfolio ledger for per-sleeve P&L.

4.3 — Market data & backtest

Market Data is the single gateway to Alpaca and the system of record for price history. Historical bars live in a shared TimescaleDB hypertable; real-time trades/quotes/bars stream from Alpaca's WebSocket and fan out to clients, hottest values cached in Redis. No other service calls Alpaca for data directly.

Backtest fetches bars from Market Data over gRPC and runs the compiled strategy bar-by-bar with slippage and commission. It computes return, Sharpe, Sortino, max drawdown, win rate, profit factor and an equity curve — vs a benchmark (SPY default) for alpha/beta. Jobs run on Celery; progress streams to the UI.

4.4 — The Alpaca surface (everything via libs/alpaca)

Every Alpaca call — REST and WebSocket — goes through the shared llamatrade_alpaca library; no service talks to Alpaca directly or loads credentials itself. paper=True selects the paper environment (paper-api.alpaca.markets vs api.alpaca.markets; market data from data.alpaca.markets).

Strategy DSL & Compiler

The DSL is an allocation language, not a signal language — a strategy computes target portfolio weights (e.g. {VTI: 60%, BND: 40%}), never share counts. It is an S-expression with a lossless round-trip between text, AST, JSON, and visual blocks.

example — sector rotation (regime filter + momentum top-N)(strategy "Sector Rotation" :rebalance monthly :benchmark SPY (if (> (price SPY) (sma SPY 200)) ; bull regime? (filter :by momentum :select (top 3) :lookback 90 ; top 3 sectors (weight :method momentum :lookback 90 (asset XLK) (asset XLF) (asset XLE) (asset XLV) (asset XLI))) (else (weight :method equal (asset BND) (asset GLD))))) ; defensive

Six block types

strategyRoot container (exactly one); name, :rebalance, :benchmark(strategy "…" :rebalance … children)
assetLeaf — the only construct that holds capital(asset SYM :weight n)
weightAllocation engine — assigns each child a fraction by :method(weight :method m :lookback d :top n …)
groupTransparent named container (readability/UI)(group "…" :weight n …)

Conditions

  • Comparison > < >= <= = != — true every bar it holds
  • Crossover crosses-above/below — true only on the transition (needs 2 bars)
  • Logical and / or / not
Two clocks — the behavior-defining gate

Conditions are evaluated every bar but only acted on at the :rebalance cadence (daily … annually; finest grain is daily). A crosses-above that fires between rebalance dates of a monthly strategy is missed — prefer state comparisons (>) for low-frequency strategies.

Authoring tips

  • Start static, then add logic — begin with specified/equal, layer in if/filter once it works.
  • Match cadence to logic — comparisons for low-frequency rebalances; crossovers shine on daily.
  • Always set a :benchmark so reporting can compute alpha/beta.
  • Mind the warm-up — (sma SPY 200) needs ≥ 200 bars before it produces signals.
  • Diversify the universe for dynamic methods — momentum / inverse-vol need several uncorrelated candidates.
  • Target weights are per-sleeve, not your whole account — see §06.

The Portfolio Ledger

The hard problem: one Alpaca account serves multiple strategies + manual trades, but the broker only knows aggregates (one position per symbol, one cash balance). The ledger gives every holding provenance, prevents strategies from interfering, and accounts each strategy's capital correctly — via an event-sourced, double-entry book (the UMA / overlay-manager pattern).

Account → virtual sleeves → provenance-bearing lots

Sleeves
Unallocated

type=strategy · alloc_capital · cash + lots
Lots
LOT — provenance-bearing
symbol · qty · cost_basis · opened_by_order · opened_at / closed_at  |  a user "holding" = Σ lots across sleeves
Invariants (continuously assertable): (1) POSITIONS: Σsleeves sleeve_qty(symbol) == broker_qty(symbol) ∀ symbol (2) CASH: Σsleeves sleeve_cash == broker_cash (3) SLEEVE: sleeve.equity = sleeve.cash + Σ(lot market value)

Double-entry means every share/cash movement has matched legs summing to zero, so conservation is a checksum — drift is detectable, never silent. Projections (sleeve positions, cash, P&L, account aggregate, holding history) are pure folds of the log and cannot diverge; crash recovery is a replay.

Cash mechanics — one broker pool, per-sleeve virtual balances

  • Allocate (Unallocated → strategy sleeve) — emits CapitalAllocated after checking Unallocated.free ≥ X. Purely virtual; no broker movement.
  • Reserve → debit → release — a buy reserves est. cost on submit, converts it to a real debit (qty×price + fees) on fill, and releases it on cancel/reject; sells credit the sleeve.
  • Transfer (sleeve → sleeve) — virtual if free cash covers it; if cash is tied up in positions the coordinator raises cash (sells in the source sleeve) first.
  • Deposit / withdraw — deposits land in Unallocated; withdrawals draw free cash, liquidating if short.
  • Dividends / fees — attributed to the lot owner; a dividend on a shared symbol splits pro-rata by lot quantity across the sleeves holding it.

Settlement is a real fork: cash accounts track settled vs unsettled (T+1) to respect good-faith / free-ride rules; margin accounts rotate same-day on buying power.

Worked example — provenance after a regime flip

One broker position per symbol, but the ledger keeps every lot's origin. After Strategy B rotates risk-off into TLT that Strategy A already holds, the user-facing holding history stays a faithful audit trail:

holding history — TLT · total 327.1 shares · one broker position, three attributed lotsTLT ├─ Strategy A 126.32 sh bought Jan 2 @ $95.00 (initial all-weather) ├─ Strategy A 14.27 sh bought Apr 1 @ $100.00 (quarterly rebalance add) └─ Strategy B 186.50 sh bought Apr 1 @ $100.00 (regime flip → risk-off)
Sizing against the sleeve — why strategies don't collide

An intended order is the delta between target and the sleeve's own holdings: target_value = sleeve_equity × (weight/100); sells reduce that sleeve's lots (FIFO). Sizing against sleeve equity, not account equity is exactly what lets many strategies share one account without double-counting capital. A drift tolerance (e.g. 5%) avoids churn.

Block-and-allocate execution

When multiple sleeves trade the same symbol in one cycle, the coordinator doesn't fire competing orders — it follows the institutional block-and-allocate pattern:

1
Bunch
net all sleeve intents for a symbol into one block (A +5, B −50 → broker −45)
›
2
Submit
a single block order to the broker
›
3
Allocate
fills back to sleeves at VWAP; offsetting intents become an internal cross

Gross execution (separate orders) makes attribution trivially 1 order ↔ 1 sleeve; netting / internal-cross adds proportional fill-splitting — where the subtle accounting bugs live — so it graduates from gross only once the ledger is proven.

Ownership: Portfolio (:8860) owns the ledger, allocation brain, projections, and reconciliation (book of record). Trading (:8850) is the execution arm — risk checks, block orders, fills. Fills flow Trading → Portfolio; intended orders flow Portfolio → Trading.

Idempotent submission & crash recovery

Orders use a deterministic client_order_id derived from the signal, so the same signal always yields the same id and Alpaca treats it as an idempotency key — a retry after a crash collapses onto the same broker order instead of placing a second one.

deterministic client_order_id (also the ledger attribution key)data = f"{session_id}:{symbol}:{side}:{signal_timestamp.isoformat()}" client_order_id = "lt-" + hashlib.sha256(data.encode()).hexdigest()[:16]

On restart the runner re-syncs equity, reconciles positions against the broker (source of truth), and re-warms indicators from market data. An order recorded but not persisted before a crash is rejected as a duplicate at the broker instead of double-filling.

Retired

The trading service previously carried a separate session-level event-sourcing subsystem (trading_events, EventSourcedOrderExecutor, SessionState replay). It was never wired into the live path and has been retired — the portfolio ledger is the event-sourced book of record.

10

Event Fabric

Real-time and async delivery runs on Redis Streams (pub/sub fully retired). The new libs/events wraps a proto envelope + pluggable transport + a typed catalog, so producers/consumers swap transports (Redis Streams today, Kafka later) without changes.

Two delivery modes

  • Tail read — live UI/gRPC clients read from a cursor (XREAD); each sees the full stream and a reconnect replays the gap (no lost fills).
  • Consumer group — durable competing consumers (ledger ingestion) via XREADGROUP + XACK + XAUTOCLAIM; each group gets every message once, unacked entries reclaimable on restart.

Idempotency & outbox

  • Stable Event.id; ledger dedups on event_id = sha256(client_order_id).
  • At-least-once delivery + dedupe = effectively exactly-once.
  • Trading's trading_events table is the producer outbox; a periodic sweep backstops any stream loss — self-healing.
